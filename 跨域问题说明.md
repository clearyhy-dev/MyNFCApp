# 跨域问题分析

## 问题原因

### 1. 跨域限制不是证书问题
- 跨域限制是浏览器的**同源策略（Same-Origin Policy）**
- 即使HTTPS证书完全正确，只要协议、域名、端口有一个不同，就属于跨域
- 主窗口：`file://` 或 `app://` 协议（Cordova应用）
- H5页面：`https://49.232.195.86`
- **这是跨域的，与证书无关**

### 2. 为什么直接调用cordova.js不成功

**Cordova exec bridge的工作原理：**
```
JavaScript (cordova.exec)
    ↓
WebView JavaScriptInterface (通过prompt)
    ↓
原生Android代码
```

**关键限制：**
- Cordova的`exec` bridge依赖于WebView的`JavaScriptInterface`
- 这个bridge**只在主窗口的WebView中有效**
- **iframe中的页面无法直接使用主窗口的exec bridge**
- 即使能访问到`window.parent.cordova.exec`，调用也无法到达原生代码

### 3. 为什么iframe中无法直接调用

**技术原因：**
1. **exec bridge是绑定到WebView的**：Cordova的exec通过`prompt()`调用JavaScriptInterface，这个接口只在主窗口的WebView中注册
2. **iframe有自己的执行上下文**：iframe中的页面有独立的JavaScript执行环境
3. **原生bridge无法跨iframe**：原生代码的JavaScriptInterface无法被iframe中的页面调用

## 解决方案

由于必须使用远程H5页面，且不能使用本地页面，唯一的可行方案是：

### 方案：透明桥接（让H5页面看起来像直接调用）

H5页面中加载一个"代理"的cordova.js，它：
- 暴露与真实cordova.js相同的API
- 底层通过postMessage与父窗口通信
- 对H5页面代码完全透明

这样H5页面可以：
```javascript
// H5页面代码，看起来就像直接调用cordova.js
cordova.plugins.NFCLockPlugin.init(success, error);
```

但实际上底层是通过postMessage桥接到主窗口的。

## 当前实现

当前代码已经实现了这种透明桥接：
- H5页面：使用`cordova.plugins.NFCLockPlugin.init()`（看起来像直接调用）
- 底层：通过postMessage桥接到主窗口
- 主窗口：使用真正的cordova.exec调用原生代码

这就是为什么必须使用桥接的原因 - 技术限制，不是设计问题。
